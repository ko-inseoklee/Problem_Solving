# B-Tree

## 1. 탄생 배경

### 1) 왜 BST나 AVL이 아닌 B-Tree가 탄생했을까?
    - HDD에서 데이터를 찾을 때, 노드를 가르키는 포인터를 찾기 위해 I/O 오버헤드가 발생한다.
    - BST의 경우에는 Worst case 시 Skewed 되어 연결리스트와 비슷한 형태로 동작하고, O(n)이 소모된다.
    - AVL의 경우에는 BST를 보완하지만, 자식, 부모 노드간의 물리적 거리가 가깝다고 보장할 수 없기 때문에 연관 데이터를 인접 공간에서 불러올 수 없다 => I/O 오버헤드 발생
    - 노드의 개수는 동일한데, 저장 측면에 어떤 차이가 있을까?
        - AVL은 한개의 노드에 자식노드의 포인터를 두개 갖고 있고, 자식노드의 포인터는 다른 디스크(또는 SSD 페이지)에 있는 경우가 빈번하기 때문에, 결국 다른 위치의 데이터를 탐색하는 I/O 오버헤드가 발생한다. 같은 페이지 내에 연관 노드를 묶어놓는 것이 I/O 오버헤드 측면에서 효율적이다.
